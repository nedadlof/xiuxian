# 修仙宗门放置游戏 Web 原型架构

## 1. 目标

- 依据 `修仙宗门放置游戏_PRD_v2.0`、`资源系统专项PRD`、`弟子堂PRD`、`藏经阁PRD`、`太古远征PRD`、`万象森罗录PRD` 实现可玩的网页原型。
- 优先做出“可循环、可扩展、可保存”的单机放置体验，而不是一次性堆满全部终局内容。
- 所有核心系统通过注册表与配置驱动，方便后续新增职业、地图、科技节点、气运之子、凶兽与事件。

## 2. MVP 范围

### 已纳入

- 资源生产循环：灵石、灵木、灵草、灵铁、丹药、符箓、道蕴
- 建筑与人口：6 类产业、宿舍、工人分配、离线收益
- 藏经阁：纪元式解锁、科技节点、全局修正
- 战争：雇佣兵训练、基础职业克制、阵型、地图关卡、战报与掉落
- 弟子堂：若干气运之子、驻守/出征切换、基础长老效果
- 万象森罗录：首批凶兽与系统联动接口
- UI：多 Tab 单页应用、日志、存档、摘要面板

### 延后至扩展包

- 15 职业完整实装
- 七卷 108 凶兽完整图鉴
- 长线剧情演出、周常残局、肉鸽探索、跨服/联盟内容
- 数值重平衡工具与批量编辑器

## 3. 模块分层

### 核心层 `src/core`

- `eventBus.js`：跨模块事件总线
- `store.js`：统一状态容器、订阅、事务式更新
- `engine.js`：Tick 驱动、离线结算、系统注册
- `save.js`：本地存档/读档/版本迁移
- `registry.js`：统一插件注册表，供职业、科技、凶兽、弟子扩展

### 数据层 `src/data`

- `balance.js`：数值常量与曲线公式
- `buildings.js`：建筑定义
- `techTree.js`：藏经阁节点
- `units.js`：雇佣兵职业、克制与阵型槽
- `disciples.js`：气运之子/长老定义
- `beasts.js`：凶兽定义与触发效果
- `stages.js`：战争关卡配置

### 系统层 `src/systems`

- `economySystem.js`：资源产出、分配、建筑升级、维护费
- `scriptureSystem.js`：道蕴产出、节点解锁、纪元推进
- `warSystem.js`：训练、编队、出征、结算、掉落
- `discipleSystem.js`：角色获得、驻守、出征增益
- `beastSystem.js`：凶兽解锁、被动生效、联动效果
- `logSystem.js`：统一日志与摘要事件

### 表现层 `src/ui`

- `layout.js`：整体页面布局与导航
- `panels/*.js`：各 Tab 面板
- `components/*.js`：资源卡、按钮、列表、战报、提示条
- `theme.css`：主题与响应式样式

## 4. 模块实现顺序

1. 核心层：状态、事件、Tick、存档、插件接口
2. 数据层：建筑/科技/战斗/弟子/凶兽配置
3. 经济系统：资源、建筑、宿舍、工人、离线收益
4. 藏经阁系统：道蕴、节点、纪元推进
5. 战争系统：兵种、编队、关卡、战报、掉落
6. 弟子堂 + 万象：驻守/出征与被动联动
7. UI 整合：Tab、面板、状态展示、操作反馈

## 5. 扩展接口设计

### 5.1 注册表驱动

- 新科技：向 `techTree` 注册新节点即可，效果通过 `modifierHooks` 注入。
- 新兵种：向 `unitRegistry` 注册职业、基础属性、克制标签、技能处理器。
- 新弟子：向 `discipleRegistry` 注册角色定义与效果函数。
- 新凶兽：向 `beastRegistry` 注册解锁条件与被动效果。
- 新关卡：向 `stageRegistry` 追加地图节点，无需改动核心战斗流程。

### 5.2 修正器管线

- 各系统统一通过 `getModifiers(state, context)` 汇总加成。
- 修正器来源可叠加：科技、弟子、凶兽、阵型、事件、未来神器。
- 公式层只读配置，不直接耦合 UI。

### 5.3 摘要驱动开发

- 每完成一个模块，立即把以下信息写入 `docs/module-summary.md`：
  - 责任范围
  - 暴露接口
  - 状态切片
  - 事件输入/输出
  - 扩展点
- 下一个模块实现前必须先读取摘要文件，按已落地接口对接。

## 6. 数值策略

- 文档主数值优先：
  - 建筑升级成本采用 `基础成本 × 1.85^(n-1)`
  - 建筑产出采用 `基础产率 × 1.20^(n-1)`
  - 工人超过 10 人后进入递减收益
  - 藏经阁道蕴采用近似 `5 × 人数² / 小时`
- 补充互联网通用放置设计原则：
  - 奖励增长需快于“体感停滞”，但慢于成本膨胀
  - 离线收益要稳定、可预期、上限明确
  - 增长曲线前期强反馈、中后期靠系统联动破局
